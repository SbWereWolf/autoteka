# Руководство оператора ПК (Clerk) по данным каталога и админке MoonShine

**Дата актуализации: 2026-03-26.**

Тип документа: **runbook** для оператора контента. Инфраструктура, URL в конкретной среде,
сидер первого администратора и переменные окружения описаны в
[ADMIN_MANUAL.md](ADMIN_MANUAL.md) (раздел про back office MoonShine).

Документ опирается на код ресурсов MoonShine в
`backend/apps/ShopOperator/app/MoonShine/Resources` и обработчик
`SaveShopResourceHandler`.

---

## 1. Зона ответственности

Оператор (clerk) отвечает за:

- корректность справочников (города, категории, фичи, типы контактов);
- корректность карточек магазинов и связей с справочниками (включая
  публикацию связей «магазин — категория / фича»);
- проверку медиа и контактов перед публикацией;
- подготовку рекламных акций магазина;
- подтверждение, что данные отображаются во front office;
- контроль, что каталог, overlay и карточка магазина соответствуют
  текущему дизайну.

## 2. Вход в админку MoonShine

### 2.1. Что понадобится

- Развёрнутый **ShopOperator** с настроенной базой и учётной записью
  администратора (см. [ADMIN_MANUAL.md](ADMIN_MANUAL.md)).
- Браузер.

### 2.2. Адрес страницы входа

Префикс панели задаётся переменной `MOONSHINE_ROUTE_PREFIX` (по умолчанию
`admin`). Страница входа: **`/admin/login`** относительно базового URL
приложения ShopOperator.

Пример из локальной среды (как в [ADMIN_MANUAL.md](ADMIN_MANUAL.md)):

- `http://127.0.0.1:8081/admin/login`

В production путь тот же относительно домена, где опубликован ShopOperator:
`/admin/login`.

### 2.3. Пошагово: войти

1. Откройте в браузере URL входа (п. 2.2).
2. В поле **Email** введите, например: `admin@example.com`  
   (значение из `MOONSHINE_ADMIN_EMAIL` в окружении ShopOperator,
   см. `backend/apps/ShopOperator/example.env`).
3. В поле **Password** введите пароль из `MOONSHINE_ADMIN_PASSWORD`
   (в примере окружения: `admin12345`).
4. Подтвердите вход (кнопка отправки формы; подпись зависит от локали
   интерфейса MoonShine).

**Ожидаемый результат:** открывается панель MoonShine с боковым меню.
Заголовки стандартных пунктов MoonShine (пользователи, роли) могут быть
на языке `APP_LOCALE` приложения. Кастомные группы меню на русском:
**Справочники**, **Данные** (см. п. 7.1).

### 2.4. Язык и время в формах

Для страниц админки задан `lang="ru"` на уровне вёрстки: нативные поля
времени (`input type="time"`) в расписании магазина отображаются в
24-часовом виде без AM/PM (при типичных настройках браузера).

## 3. Учётные записи администраторов MoonShine

Раздел относится к встроенным сущностям MoonShine: **пользователи** и
**роли**. В меню они обычно находятся в системной группе (название
зависит от перевода MoonShine, при английской локали — например
**Admins**, **Roles**).

### 3.1. Список администраторов

На странице списка доступны:

- **Фильтры:** по роли и по E-mail.
- **Выбор колонок** таблицы (column selection).
- Колонки включают **Обновлён** (дата и время обновления записи).

### 3.2. Создание нового администратора (пошагово)

1. Откройте ресурс пользователей MoonShine в меню.
2. Нажмите действие **создания** новой записи (кнопка с иконкой «плюс»
   или подпись вроде **Add** / **Create** при английской локали).
3. Вкладка **основной информации** (название вкладки зависит от перевода):
   - **Роль** — выберите существующую роль (**обязательное поле**).
     При необходимости можно создать роль из связанного интерфейса
     (**creatable**).
   - **Имя** — например: `Иван Оператор` (**обязательно**).
   - **Email** — например: `operator@example.com` (**обязательно**,
     уникальный).
   - **Аватар** — по желанию (форматы: jpeg, jpg, png, gif).
4. Вкладка **пароля**:
   - При **создании** пользователя пароль **обязателен**; укажите пароль
     и **повтор** в двух полях (требования — как у
     `Password::defaults()` в Laravel).
5. Сохраните форму.

**Ожидаемый результат:** новая запись появляется в списке; вход возможен
с указанным email и паролем (если политика MoonShine это допускает).

### 3.3. Роли

Ресурс ролей использует **модальные окна** для создания и редактирования.
Имя роли **обязательно**, минимальная длина **5 символов**.

## 4. Справочники

В меню группа **Справочники** содержит:

| Пункт меню        | Сущность        |
|-------------------|-----------------|
| **Города**        | города          |
| **Категории**     | категории       |
| **Фичи**          | фичи (особенности) |
| **Типы контактов**| типы контактов  |

### 4.1. Общие правила для всех справочников

**Страницы:** список, **форма редактирования**, **детальная страница**
(просмотр без перехода к полям ввода в стиле формы — набор полей как
в списке/сводке).

**Удаление:** из интерфейса **нельзя** удалить запись справочника и нет
массового удаления (действия отключены в коде ресурса).

**Поля формы:**

| Поле в интерфейсе | Обязательность | Смысл |
|-------------------|----------------|-------|
| **Код**           | только просмотр | Технический код для API и URL. В форме и списке отображается как **неизменяемое** поле (preview). Для **новой** записи до первого сохранения может быть пустым; после сохранения заполняется **автоматически** из **Название** (латиница, цифры, `-`, `_`; при коллизии добавляется суффикс). |
| **Название**      | да             | Человекочитаемое имя. |
| **Sort**          | да, число ≥ 0  | Порядок сортировки. По умолчанию подставляется **следующий** свободный шаг: максимум `sort` в таблице + **10**. |
| **Опубликован**   | переключатель | Если выключено, запись **не попадает** в публичные списки API витрины (`*-list` и связанная выдача). |

После **сохранения** справочной записи выполняется переход на **список**
(index), а не на детальную страницу.

**Снятие с публикации (пошагово):**

1. Откройте нужный справочник (**Справочники → …**).
2. Откройте запись на **редактирование** (иконка карандаша / **Edit**).
3. Переключите **Опубликован** в выключенное состояние.
4. Сохраните форму.

**Ожидаемый результат:** в списке у записи отображается неактивная
публикация; API витрины не отдаёт эту строку в публичных списках.

### 4.2. Пример: город

1. **Справочники → Города** → действие создания записи.
2. **Название:** `Новосибирск` (плейсхолдер в форме: «Например:
   Новосибирск»).
3. **Sort:** оставьте предложенное значение или введите своё (целое ≥ 0).
4. **Опубликован:** включено.
5. Сохраните.

После сохранения в форме и в списке появится **Код** (например
`novosibirsk` или с суффиксом при коллизии).

### 4.3. Пример: категория

1. **Справочники → Категории** → создать.
2. **Название:** `Шины и диски` (плейсхолдер: «Например: Шины и диски»).
3. **Sort** и **Опубликован** — как в п. 4.1.
4. Сохраните.

### 4.4. Пример: фича

1. **Справочники → Фичи** → создать.
2. **Название:** `Шиномонтаж` (плейсхолдер: «Например: Шиномонтаж»).
3. **Sort** и **Опубликован** — как в п. 4.1.
4. Сохраните.

### 4.5. Пример: тип контакта

1. **Справочники → Типы контактов** → создать.
2. **Название:** `Телефон` (плейсхолдер: «Например: Телефон»).
3. **Sort** и **Опубликован** — как в п. 4.1.
4. Сохраните.

## 5. Магазины: сквозной пример «от справочников до магазина»

Меню: группа **Данные → Магазины**.

### 5.1. Подготовка (уже сделано в §4)

Для примера ниже должны существовать **опубликованные**:

- город **Новосибирск**;
- категория **Шины и диски**;
- фича **Шиномонтаж**;
- тип контакта **Телефон**.

Если вы создавали другие названия — подставьте их в шагах выбора в
выпадающих списках.

### 5.1.1. Акции магазина

Акции не создаются как отдельная независимая сущность из общего списка.
Рабочий путь оператора:

1. открыть detail нужного магазина;
2. перейти по ссылке `Добавить рекламную акцию`;
3. заполнить форму акции уже в контексте этого магазина.

Что важно помнить:

- код акции вводить вручную не нужно, он пересчитывается автоматически;
- акция создается только из detail нужного магазина;
- смена магазина у уже созданной акции недоступна;
- текст-only акция допустима, пустую галерею делать не нужно;
- future-акции на витрине должны быть опубликованы, иначе они не
  появятся;
- active-акции на detail магазина в админке видны отдельным блоком
  `Акции магазина`.
- в форме акции магазин показывается как `город: магазин` только для
  просмотра.

Структура данных акции:

- хранит код, название, описание, даты начала и окончания;
- имеет флаг публикации;
- галерея хранит `original_name`, сортировку и флаг публикации;
- код акции вычисляется из `shop.code + '-' + slug(title)` и при
  коллизии получает числовой суффикс.

### 5.1.2. Как создать рекламную акцию

1. Откройте `Данные -> Магазины`.
2. Найдите нужный магазин и откройте его detail-страницу.
3. В блоке `Создание рекламной акции` нажмите
   `Добавить рекламную акцию`.
4. Убедитесь, что сразу открылось редактирование уже созданной акции.
5. Проверьте автозаполненные значения:
   - `Название` по шаблону `магазин YYYY-MM-DD HH:mm`;
   - `Описание` по шаблону `рекламная акция магазин`;
   - `Дата начала` и `Дата окончания` равны `сегодня + 7 дней`;
   - `Опубликована` выключена.
6. При необходимости отредактируйте текст, даты и публикацию.
7. При необходимости добавьте изображения в `Галерея`; текст-only
   акция допустима.
8. Сохраните форму.

Ожидаемый результат:

- акция сохраняется за текущим магазином;
- код пересчитывается автоматически из кода магазина и названия акции;
- при коллизии кода backend сам добавляет следующий числовой суффикс;
- detail магазина показывает новую акцию в блоке `Акции магазина`.

### 5.1.3. Demo-команды для clerk-сценария

Для быстрого наполнения базы демонстрационными акциями в ShopOperator
доступны две artisan-команды:

```bash
cd backend/apps/ShopOperator
php artisan demo:promo:purge
php artisan demo:promo:create
```

Порядок обязателен:

1. сначала `demo:promo:purge`;
2. затем `demo:promo:create`.

Что делает `demo:promo:create`:

- случайно распределяет магазины по четырём группам;
- для части магазинов не создаёт акции;
- для остальных создаёт past/future/active promotions;
- для future-only и past-only групп создаёт от 1 до 5 акций;
- для mixed-группы создаёт от 1 до 5 акций, где категории выбираются
  независимо с вероятностью 85%, а fallback при пустом выборе —
  `active`;
- для каждой акции независимо решает, будет ли она опубликована, с
  вероятностью 75%;
- копирует уже существующие изображения магазинов во временную папку
  ОС и добавляет их в promotion gallery под новыми именами.

Что делает `demo:promo:purge`:

- удаляет файлы promotion gallery через текущий admin save-flow;
- затем удаляет все promotions через SQL.
- shop_id для purge собирается только через grouped join
  `promotion_gallery_image -> promotion`.

Важно:

- `demo:promo:create` не работает поверх непустой таблицы promotions и
  завершится ошибкой с подсказкой сначала выполнить purge;
- `demo:promo:purge` может завершиться с кодом `1`, если после стадии
  очистки gallery были обнаружены residual-строки. В этом случае rollback
  не делается: команда всё равно доводит purge до конца и выводит
  предупреждение.
- demo-команды используют temp dir ОС для staging изображений; временные
  файлы удаляются после выполнения.
- перед изменениями схемы или migration, затрагивающими promotions,
  операторская и инженерная проверка выполняется по
  [TESTING.md](TESTING.md) с обязательным циклом
  `backup -> migrate -> rollback -> migrate`.

### 5.2. Создать магазин

1. **Данные → Магазины** → создание новой записи.
2. **Код** — только просмотр: для новой записи пусто до сохранения;
   после сохранения заполнится автоматически (как у справочников).
3. **Название** (обязательно): например `АвтоТека на Гоголя`.
4. **Sort** (обязательно): число ≥ 0. После выбора **Город** значение по
   умолчанию пересчитывается как «максимум `sort` среди магазинов этого
   города + 10»; при необходимости измените вручную.
5. **Город** (обязательно): выберите **Новосибирск** (поиск по списку
   доступен).
6. **Опубликован:** включите, если магазин должен попасть в витрину.
7. **Описание:** например `Запчасти, масла, аксессуары. Доставка по городу.`
8. **Ссылка на сайт:** `https://example.com` (пустое значение
   допустимо).
9. **Слоган:** например `Всё для вашего авто`.
10. **Широта / Долгота:** например `55.0287` и `82.9235` или оставьте
    пустыми (если заполнено — должно быть числом).
11. **Текст расписания:** например `Воскресенье — выходной`.
12. **Логотип:** при необходимости загрузите файл (расширения: jpg, jpeg,
    png, webp). Старое изображение с диска удаляется при замене.

**Блок «Категории» (повторяемые строки):**

1. Добавьте строку (**добавить элемент** / иконка «+» — формулировка
   зависит от MoonShine).
2. **Категория:** выберите **Шины и диски**.
3. **Опубликован** (на строке связи): **включено** — связь магазина с
   категорией участвует в выдаче каталога при прочих условиях.
4. При необходимости добавьте ещё строки; дубликаты одной и той же
   категории схлопываются при сохранении до одной связи.

**Блок «Фичи»:** аналогично — выберите **Шиномонтаж**, переключатель
**Опубликован** на строке **включён**.

**Блок «Контакты»:**

1. Добавьте строку.
2. **Тип:** **Телефон**.
3. **Значение:** например `+7 913 000-00-01` (**обязательно**).
4. **Sort** (обязательно): например `10` (по умолчанию может подставиться
   следующий порядок для этого магазина).
5. **Опубликован:** включено.

Нельзя сохранить два контакта с одним и тем же **типом** и **одинаковым
значением** (без учёта регистра) — сервер вернёт ошибку валидации.

**Блок «Галерея»:**

- Для каждой строки загрузите **Файл** (те же расширения, что у логотипа),
  задайте **Sort** (обязательно) и **Опубликован**.
- Строка без файла при сохранении не создаёт запись в галерее.

**Блок «Расписание по дням»:**

- Можно добавить до **7** строк (по дням недели).
- **День недели**, **Начало**, **Конец** — обязательны; время вводится
  нативным полем времени, шаг **15 минут** (900 секунд).
- **Sort** обязателен; **Опубликован** управляет показом интервала.
- Если для одного дня недели передано несколько строк, при сохранении
  фактически сохраняется одна строка на день (по логике синхронизации на
  сервере).

**Сохранение:** нажмите кнопку сохранения формы (подпись зависит от
локали MoonShine, часто **Save** / **Сохранить**).

**Ожидаемый результат:** выполняется **редирект на детальную страницу**
магазина. На ней отображаются: код, основные поля, логотип, текстовые
сводки по категориям и фичам (с пометкой «не опубликовано», если связь
снята с публикации), контакты, расписание, блок **Галерея** с миниатюрами
в ряд; **клик по миниатюре** открывает всплывающий просмотр изображения.

## 6. Магазины: изменение записи, вложенные поля, публикация

### 6.1. Как открыть редактирование

- Из списка **Магазины** — действие **редактирования**;
- или с **детальной** страницы магазина — переход к форме (как предлагает
  MoonShine).

### 6.2. Основные поля

Назначение полей совпадает с п. 5.2. **Название**, **Город** и **Sort**
обязательны. **Код** по-прежнему только для просмотра: смена названия
не перезаписывает код автоматически после того, как код уже задан.

### 6.3. Снять с публикации сам магазин

На форме переключите **Опубликован** в выключенное состояние и сохраните.
Непубликованный магазин не попадает в публичный каталог города.

### 6.4. Категории и фичи: публикация связи

Даже если справочник **Категория** / **Фича** опубликованы, витрина
учитывает отдельно переключатель **Опубликован** на **строке связи** в
форме магазина. Если его выключить, категория или фича **не попадут** в
состав выдачи для этого магазина (фильтр `pivot.is_published` в API
каталога).

**Пошагово:** откройте форму магазина → найдите строку в блоке
**Категории** или **Фичи** → выключите **Опубликован** на этой строке →
сохраните → откройте детальную страницу и проверьте пометку
«не опубликовано» рядом с названием.

### 6.5. Удалить строку вложенного блока

У блоков **Категории**, **Фичи**, **Контакты**, **Галерея**,
**Расписание** доступно удаление строки (действие удаления элемента
списка в UI). Для **галереи** файл удалённой строки **удаляется с диска**
при сохранении магазина.

### 6.6. Изменить значение контакта или порядок

Откройте форму, измените **Значение** и/или **Sort**, сохраните. Помните
про **уникальность** пары тип + значение среди всех строк контактов
магазина.

## 7. Прочие возможности и особенности админки

### 7.1. Меню

- **Справочники** — города, категории, фичи, типы контактов.
- **Данные** — магазины.
- Далее — системные разделы MoonShine (пользователи, роли и т.д.).

### 7.2. Поиск и сортировка

- В списках доступен **поиск** по полям, заданным в ресурсе (в т.ч. по
  **коду** для магазинов и справочников).
- Часть колонок поддерживает **сортировку** по клику на заголовок.

### 7.3. Детальная страница магазина

Используйте для **обзора** сохранённых данных и проверки галереи без
редактирования. Переход из списка — действие **просмотра** / переход на
**Detail** (зависит от темы MoonShine).

### 7.4. Поведение после сохранения

- **Справочник** → возврат к **списку**.
- **Магазин** → переход на **детальную страницу** этого магазина.

### 7.5. Валидация при сохранении магазина (сервер)

Обработчик `SaveShopResourceHandler` при сохранении проверяет в том числе:

- **Название** и **Город** заполнены; город существует в базе.
- **Sort** задан, неотрицателен.
- **Широта** и **Долгота** — либо пусто, либо число.
- Для **каждой** строки блоков **Категории** и **Фичи**, где выбрана
  категория/фича, в данных запроса должен присутствовать ключ
  **Опубликован** (`is_published`); иначе сервер вернёт ошибку с текстом
  про необходимость явно указать «Опубликован».

Синхронизация связей магазина с категориями и фичами сохраняет флаг
**Опубликован** на **связи** (pivot), отдельно от публикации записи
справочника.

### 7.6. Витрина и API (кратко)

В выдаче каталога города участвуют только:

- опубликованный **город**;
- опубликованные **магазины**;
- для каждого магазина — **категории и фичи**, у которых опубликована
  сама сущность справочника **и** опубликована **связь** с магазином.

Подробнее про проверку снаружи — в §9 и в [ADMIN_MANUAL.md](ADMIN_MANUAL.md).

## 9. Критерии качества публикации

После публикации:

- список городов/категорий/фич доступен через API `/api/v1/*-list`;
- карточка магазина открывается по `/shop/:code`;
- слоган, контакты и `shopScheduleNote` отображаются в карточке;
- изображения доступны по `/storage/*` и не отдаются как SPA HTML;
- в MoonShine на **детальной** странице магазина отображаются актуальные
  данные и миниатюры галереи; клик по миниатюре открывает просмотр.

Если картинка в браузере пустая, а запрос к `/storage/*` возвращает
`200 text/html`, проблема не в данных магазина, а в серверной раздаче
public media (см. [ADMIN_MANUAL.md](ADMIN_MANUAL.md)).

## 10. Ручная проверка сценария в админке (чеклист)

Рекомендуется **обязательно** пройти в браузере на развёрнутом ShopOperator
(см. запуск в [ADMIN_MANUAL.md](ADMIN_MANUAL.md)). Ниже — контрольный
список; шаги выше в §2–§7 согласованы с текущим кодом ресурсов и
обработчика сохранения.

1. Вход (§2).
2. Справочник: создать запись, открыть **деталь**, снять и вернуть
   **Опубликован**.
3. Магазин: создать по §5, убедиться в редиректе на **деталь** и в
   работе **галереи** (клик по миниатюре).
4. У одной строки **Категории** или **Фичи** снять **Опубликован**
   (связь), сохранить; проверить витрину или API, что признак не попал в
   выдачу.
5. Поля времени в расписании отображаются в 24-часовом виде.

Если подписи кнопок отличаются от ожидаемых, ориентируйтесь на описанные
**поля форм** (они на русском в ресурсах приложения) и на иконки
действий MoonShine.
