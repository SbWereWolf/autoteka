# API Contract v1 (frontend ↔ backend) — Autoteka

**Дата:** 2026‑03‑04  
**Цель:** зафиксировать сигнатуры эндпоинтов и объектную модель данных
под текущий фронт (переход с моков на API), без кода и без
проектирования БД.

---

## 1) Базовые принципы (зафиксировано)

- **ID-формат**: для клиента это **opaque string** (может быть
  “barnaul”, “123”, UUID и т.п.). Клиент не интерпретирует ID.
- **Стабильный порядок**: во всех list‑эндпоинтах есть поле `sort`.  
  Дефолтный выбор = **первый элемент** списка, **после** сортировки по
  `sort ASC` (и стабильного tie‑break, например `id ASC`).
- **Пустые списки**: `200 OK` + пустой массив. Фронт:
  - дизейблит селекты/чипсы,
  - не выполняет сортировки/фильтрации (или работает с пустыми
    наборами),
  - если нет городов → нет магазинов.
- **Локализация**: всегда RU (один язык).
- **Медиа**: сервер возвращает **относительные URL**.
  **root‑relative** (начинается с `/`), чтобы корректно резолвилось на
  любых роутингах.

> Настройки презентации/выборы пользователя — остаются на фронте
> (`localStorage`).

## 2) Объектная модель (DTO для API v1)

### 2.1 City (для списков)

```json
{
  "id": 11,
  "code": "barnaul",
  "name": "Барнаул",
  "sort": 10
}
```

### 2.2 Category (справочник категорий)

```json
{
  "id": 22,
  "code": "cat_domestic",
  "name": "Отечественные запчасти",
  "sort": 10
}
```

### 2.3 Feature (справочник фич)

```json
{
  "id": 33,
  "code": "feat_fast_delivery",
  "name": "Самая быстрая доставка",
  "sort": 10
}
```

### 2.4 ShopPublic (магазин без контактов)

Это “полная информация для клиента”, **кроме контактов**. Используется
и в каталоге, и на странице магазина.

```json
{
  "id": 44,
  "code": "shop_001",
  "cityId": "barnaul",

  "name": "Авто‑Мир",
  "description": "…",
  "workHours": "Пн‑Пт 9:00‑18:00\nСб 10:00‑16:00",
  "siteUrl": "https://example.com",

  "thumbUrl": "/media/shops/shop_001/thumb.png",
  "galleryImages": [
    "/media/shops/shop_001/1.jpg",
    "/media/shops/shop_001/2.jpg"
  ],

  "categoryIds": ["cat_domestic", "cat_japan"],
  "featureIds": ["feat_fast_delivery", "feat_pickup"]
}
```

**Замечания:**

- `thumbUrl` и `galleryImages[]` опциональны. Если пусто/нет — на
  фронте показывается empty‑state.
- `categoryIds[]` / `featureIds[]` — **идентификаторы**, для
  отображения названий фронт использует справочники `/category-list` и
  `/feature-list`.
- Сортировка каталога на фронте опирается на наличие выбранных
  `categoryIds` и `selectedFeatureId` в `featureIds`.

### 2.5 ContactsResponse (контакты, отфильтрованные по типам)

Контакты возвращаются **отдельно**, и **по типам**, причём значение
типа — массив строк.

```json
{
  "phone": ["+7 (999) 123‑45‑67", "+7 (999) 111‑22‑33"],
  "email": ["info@example.com"],
  "telegram": ["https://t.me/example"],
  "address": ["Барнаул, ул. Ленина, 1"]
}
```

> Контакт — просто строка (без имени/должности/лейбла).

## 3) Эндпоинты v1 (канонические сигнатуры)

> В URL лучше фиксировать единый стиль. Ниже — канонический вариант
> **без trailing slash**.  
> (Можно поддержать trailing slash на сервере, но в контракте лучше
> один вариант.)

---

### 3.1 CatalogPage — магазины города

**GET** `/api/v1/city/{cityId}`

**Назначение:** вернуть магазины, которые принадлежат городу.

**Query (опционально):**

- `q` — строка поиска (по имени магазина; при желании можно расширить
  на description)
- `page` — номер страницы (1‑based)
- `perPage` — размер страницы

**Response 200:**

```json
{
  "items": [
    /* ShopPublic[] */
  ],
  "page": 1,
  "perPage": 24,
  "total": 132
}
```

**Примечания:**

- Даже если фронт потом пересортирует данные по своим правилам —
  сервер должен отдавать `items` в **детерминированном порядке**
  (например, `sort ASC, id ASC`), чтобы дефолтные/тестовые сценарии
  были воспроизводимы.
- Если город существует, но магазинов нет → `items: []`, `total: 0`.

**Ошибки:**

- `404 Not Found` если `cityId` не существует.

---

### 3.2 ShopPage — данные магазина (без контактов)

**GET** `/api/v1/shop/{shopId}`

**Назначение:** получить “полную карточку” магазина для клиента **без
контактов**, включая все `categoryIds` и `featureIds` (для отображения
и/или сортировки).

**Response 200:** `ShopPublic`

**Ошибки:**

- `404 Not Found` если `shopId` не существует.

---

### 3.3 CitySelect — список городов

**GET** `/api/v1/city-list`

**Response 200:** `City[]` (отсортирован по `sort ASC, id ASC`)

**Правило дефолта:** первый элемент списка.

---

### 3.4 CategoryChips — список категорий

**GET** `/api/v1/category-list`

**Response 200:** `Category[]` (отсортирован по `sort ASC, id ASC`)

---

### 3.5 FeatureSelect — список фич

**GET** `/api/v1/feature-list`

**Response 200:** `Feature[]` (отсортирован по `sort ASC, id ASC`)

**Правило дефолта:** первый элемент списка.

---

### 3.6 Контакты по допустимым типам

**POST** `/api/v1/shop/{shopId}/acceptable-contact-types`

**Request body:** массив типов, которые фронт умеет рендерить

```json
["phone", "email", "telegram", "whatsapp", "address"]
```

**Response 200:** `ContactsResponse` (только запрошенные типы, каждый
— массив строк)

```json
{
  "phone": ["+7..."],
  "email": ["..."]
}
```

**Правила:**

- Если фронт прислал тип, которого нет на сервере → сервер **молча
  игнорирует**.
- Если у магазина есть контакты неизвестных типов → сервер **молча
  игнорирует** (допустимо логировать).
- Возвращаются **только контакты**, без метаданных.

**Ошибки:**

- `404 Not Found` если `shopId` не существует.
- Для v1 **не вводим** жёсткую валидацию типов (нет `422` только из‑за
  неизвестного типа).

---

## 4) Формат ошибок (минимально жизнеспособный)

- **Базовый** (для 404/500 и любых “неполевых” ошибок):

```json
{ "message": "Not Found" }
```

- **Валидация (если появится позже)** — стандарт Laravel:

```json
{
  "message": "The given data was invalid.",
  "errors": {
    "fieldName": ["Error 1", "Error 2"]
  }
}
```

## 5) Что потребуется поправить на фронте (концептуально)

1. `categories[]` и `features[]` в моках сейчас строки. В API v1 это:
   - `categoryIds[]`
   - `featureIds[]`
   - отображаемые названия берутся по справочникам list‑эндпоинтов.

2. В `ShopPage` нужно:
   - добавить вывод **всех категорий и всех фич** магазина (по ID →
     name),
   - заменить текущий вывод `shop.contacts` на отдельную загрузку
     через POST `acceptable-contact-types`.

3. Дефолтный город/фича:
   - теперь берутся как **первый элемент** `city-list` /
     `feature-list` (по `sort`).

## 6) Переходный вариант “как бы API”, но данные из моков (без сети)

**Цель:** фронт начинает жить через абстракцию `ApiClient`, но
реализация на первом шаге читает `src/mocks/*.json` и возвращает
`Promise.resolve(...)`.

### 6.1 Интерфейс клиента (логический)

Методы 1:1 повторяют API:

- `getCityList(): Promise<City[]>`
- `getCategoryList(): Promise<Category[]>`
- `getFeatureList(): Promise<Feature[]>`
- `getCityShops(cityId, {q,page,perPage}): Promise<{items, page, perPage, total}>`
- `getShop(shopId): Promise<ShopPublic>`
- `postAcceptableContactTypes(shopId, types: string[]): Promise<ContactsResponse>`

### 6.2 Маппинг моков → DTO

**Справочники:**

- из `dicts.json` (где `categories/features` — строки) формируется
  список объектов:
  - `id` — детерминированно (на переходный период допустимо
    `id = исходная строка` либо slug/хеш)
  - `name` — исходная строка
  - `sort` — индекс в массиве (0,1,2…)

**Магазины:**

- из `shops.json`:
  - `city` → `cityId`
  - `categories: string[]` → `categoryIds: string[]` (через мапу
    `name → id`)
  - `features: string[]` → `featureIds: string[]` (через мапу
    `name → id`)
  - остальные поля без изменений
  - **контакты**:
    - `getShop(shopId)` возвращает объект **без контактов**
    - `postAcceptableContactTypes(shopId, types[])` строит ответ,
      группируя `shop.contacts` по `type` и фильтруя по `types[]`

### 6.3 Плюсы такого подхода

- Компоненты уже используют “API‑методы”, а не прямые импорты моков.
- Переход на настоящий сервер = замена реализации клиента, без
  переписывания страниц/компонентов.

---

## 7) Открытые решения (не блокируют v1, но полезно помнить)

- **Медиа URL**: root‑relative (`/media/...`) 
- **Поиск/пагинация**: подтверждены как “да”, но точные поля поиска и
  лимиты можно уточнить перед бэком.
- **Связи фич/категорий с городом**: намеренно **не делаем** на
  гипотезе.

---

## 8) Краткая карта “UI → endpoint”

- `CitySelect` → `GET /api/v1/city-list`
- `CategoryChips` → `GET /api/v1/category-list`
- `FeatureSelect` → `GET /api/v1/feature-list`
- `CatalogPage` → `GET /api/v1/city/{cityId}?q=&page=&perPage=`
- `ShopPage` → `GET /api/v1/shop/{shopId}` +
  `POST /api/v1/shop/{shopId}/acceptable-contact-types`
